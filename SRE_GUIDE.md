## 1. Infrastructure Overview

The platform runs entirely on GCP and is managed with Terraform. There are no VMs — compute is fully serverless via Cloud Run.

```
Internet
    │
    ├── Cloud Run: ecommerce-store       (customer storefront)
    ├── Cloud Run: ecommerce-admin       (store admin panels)
    ├── Cloud Run: ecommerce-super-admin (platform ops)
    └── Cloud Run: ecommerce-api         (backend — private egress only)
                        │
                VPC Access Connector
                        │
                 Private VPC (ecommerce-builder-vpc)
                        │
              Cloud SQL: PostgreSQL 15   (private IP, no public endpoint)
                        │
                 Cloud Secret Manager   (all secrets, no env var plaintext)
                        │
                 Cloud Storage          (media uploads bucket)
```

**Design principles:**
- The database has **no public IP**. It is only reachable from Cloud Run via the VPC Access Connector and private VPC peering. There is no external attack surface on the database.
- Secrets are stored in **Cloud Secret Manager**, not as environment variables in Cloud Run revisions. The API reads them at startup via the Secret Manager API.
- All Cloud Run services use a dedicated **service account** (`ecommerce-builder-app`) with least-privilege IAM roles — not the default Compute service account.
- Terraform state is stored in a versioned **GCS bucket**, not locally.

## 3. Initial Deployment

This is a one-time process. After this, all changes go through the CI/CD pipeline.

### 3a. Bootstrap GCP

```bash
cd infrastructure/scripts
chmod +x setup-gcp.sh create-secrets.sh

# Creates TF state bucket, service accounts, generates terraform.tfvars
./setup-gcp.sh --project=YOUR_PROJECT_ID --region=us-central1
```

The script will:
- Enable bootstrap APIs
- Create the Terraform state GCS bucket
- Create the `terraform-deployer` service account with required IAM roles
- Generate `.terraform-sa-key.json` (keep this safe — add to GitHub secrets)
- Auto-generate `infrastructure/terraform/terraform.tfvars` with random secrets pre-filled

### 3b. Edit terraform.tfvars

Open `infrastructure/terraform/terraform.tfvars` and fill in the two placeholder values:
```hcl
gemini_api_key    = "YOUR_KEY_FROM_AISTUDIO"
super_admin_email = "your-email@example.com"
```

Everything else (DB password, JWT secret, encryption key) was auto-generated by the setup script.

### 3c. Apply infrastructure

```bash
cd infrastructure/terraform
terraform plan   # review — expect ~25 resources to be created
terraform apply  # takes ~10-15 min (Cloud SQL provisioning is the slow part)
```

### 3d. Build and push initial images

The CI pipeline handles this normally, but for the first deploy you need to push images manually since the Cloud Run services reference specific image tags.

```bash
export PROJECT_ID=YOUR_PROJECT_ID
gcloud auth configure-docker gcr.io

for svc in api admin store super-admin; do
  case $svc in
    api)   context="services/api" ;;
    *)     context="apps/$svc" ;;
  esac
  docker build -t gcr.io/$PROJECT_ID/ecommerce-$svc:latest $context
  docker push gcr.io/$PROJECT_ID/ecommerce-$svc:latest
done
```

### 3e. Run initial migrations and seed

```bash
# Deploy a one-off Cloud Run job to run migrations
gcloud run jobs deploy migrate-job \
  --image=gcr.io/$PROJECT_ID/ecommerce-api:latest \
  --region=us-central1 \
  --set-env-vars="DATABASE_URL=$(gcloud secrets versions access latest \
    --secret=ecommerce-builder-db-password --format='get(payload.data)' | base64 -d)" \
  --command=npx \
  --args="prisma,migrate,deploy" \
  --service-account=ecommerce-builder-app@$PROJECT_ID.iam.gserviceaccount.com

gcloud run jobs execute migrate-job --region=us-central1 --wait
```

After migrations, add GitHub secrets (see Section 4) and push to `main` to trigger the full pipeline.

## 4. CI/CD Pipeline

Two workflows in `.github/workflows/`:

### `test.yml` — runs on every PR to `main`

- Path-filtered: only tests packages that changed
- TypeScript type checks all four packages
- Full `npm run build` for each frontend (catches Next.js build errors)
- `npm audit` for high-severity CVEs
- Docker build validation (no push)
- `all-checks-pass` job acts as the branch protection gate

### `deploy.yml` — runs on every push to `main`

Job order:
```
changes → setup-gcp
               │
    ┌──────────┼───────────┬──────────────┐
 build-api  build-admin  build-store  build-super-admin
    │
  migrate
    │
 deploy-api
    │
 ┌──┴────────┬───────────────┐
deploy-admin  deploy-store  deploy-super-admin
    │
smoke-test
```

**Required GitHub repository secrets:**

| Secret | Value |
|---|---|
| `GCP_SA_KEY` | Full JSON content of `.terraform-sa-key.json` |
| `GCP_PROJECT_ID` | GCP project ID |
| `GCP_REGION` | e.g., `us-central1` |
| `GCP_SA_EMAIL` | `terraform-deployer@{project}.iam.gserviceaccount.com` |
| `DATABASE_URL` | Full connection string for migrations job |
| `TF_DB_PASSWORD` | Database password (from terraform.tfvars) |
| `TF_JWT_SECRET` | JWT secret (from terraform.tfvars) |
| `TF_ENCRYPTION_KEY` | 32-char encryption key (from terraform.tfvars) |
| `GEMINI_API_KEY` | Google AI Studio key |
| `SUPER_ADMIN_EMAIL` | Platform super admin email |

**Infrastructure changes** (edits to `infrastructure/terraform/`) trigger an additional `terraform` job at the end of the pipeline that runs `terraform plan` then `terraform apply`. This job requires the `production` GitHub environment, which you should configure with a manual approval requirement.

**Manual deploy of a single service:**

Go to GitHub → Actions → "Deploy to GCP" → "Run workflow". Set `services` to `api`, `admin`, `store`, or `super-admin`.

## 5. Monitoring & Alerting

### Cloud Monitoring (GCP native)

A Cloud Monitoring alert policy (`High Error Rate — Ecommerce Builder`) is provisioned by Terraform. It fires when the Cloud Run 5xx rate exceeds 5% over a 5-minute window.

To add a notification channel (email, PagerDuty, Slack):
```bash
gcloud alpha monitoring channels create \
  --display-name="SRE On-Call Email" \
  --type=email \
  --channel-labels=email_address=oncall@example.com
```
Then attach it to the policy in the GCP Console (Monitoring → Alerting → Policies → High Error Rate).

### Application-level anomaly detection

The API runs KL divergence checks every 60 seconds (configurable via `METRIC_SNAPSHOT_INTERVAL_MS`). Alerts fire to the `Alert` database table and appear in the super admin panel. They do **not** integrate with Cloud Monitoring by default — if you want PagerDuty/Slack integration, add a webhook call in `services/api/src/services/anomaly.ts` where the `Alert` record is created.

### Metrics tracked

| Metric name | What it measures |
|---|---|
| `error_rate` | HTTP 5xx percentage |
| `response_time_p99` | 99th percentile API response time |
| `order_count` | Orders per minute |
| `chat_sessions` | Chat sessions per minute |
| `api_requests` | Total API requests per minute |

### Cloud SQL Query Insights

Query Insights is enabled in Terraform (`query_insights_enabled = true`). Access it at:
GCP Console → SQL → `ecommerce-builder-pg` → Query Insights

This shows slow queries, top queries by CPU, and lock contention without requiring direct database access.

### Log-based metrics

All application logs are written to Cloud Logging in structured JSON format when `STRUCTURED_LOGGING=true` (set in the Cloud Run env). You can create log-based metrics in Cloud Monitoring for anything in the logs — useful for alerting on specific error messages.

```bash
# Example: count occurrences of "anomaly detected" in logs
gcloud logging metrics create anomaly-detections \
  --description="KL divergence anomalies" \
  --log-filter='resource.type="cloud_run_revision" jsonPayload.message=~"Anomaly detected"'
```

## 6. Database Operations

### Connecting to the database

The database has no public IP. You must use the **Cloud SQL Auth Proxy**:

```bash
# Install
curl -o cloud-sql-proxy https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.x.x/cloud-sql-proxy.linux.amd64
chmod +x cloud-sql-proxy

# Run (in background)
./cloud-sql-proxy --port 5432 YOUR_PROJECT_ID:us-central1:ecommerce-builder-pg &

# Connect with psql
psql "host=localhost port=5432 dbname=ecommerce_builder user=app_user password=YOUR_DB_PASSWORD"
```

Get the DB password:
```bash
gcloud secrets versions access latest --secret=ecommerce-builder-db-password
```

### Backups

Automated daily backups are configured in Terraform with 7-day retention and point-in-time recovery (PITR) enabled. Backups run at 3 AM UTC Sunday during the maintenance window.

**List available backups:**
```bash
gcloud sql backups list --instance=ecommerce-builder-pg
```

**Restore from backup:**
```bash
# This restores IN-PLACE and overwrites current data
gcloud sql backups restore BACKUP_ID \
  --restore-instance=ecommerce-builder-pg \
  --backup-instance=ecommerce-builder-pg
```

**Point-in-time recovery** (e.g., recover to state from 2 hours ago):
```bash
gcloud sql instances clone ecommerce-builder-pg ecommerce-builder-pg-recovery \
  --point-in-time="2024-01-15T10:00:00.000Z"
```

### Running migrations manually

If the CI job fails or you need to run migrations out-of-band:
```bash
gcloud run jobs execute migrate-job --region=us-central1 --wait
```

To check migration job logs:
```bash
gcloud logging read 'resource.type="cloud_run_job" resource.labels.job_name="migrate-job"' \
  --limit=50 --format="table(timestamp,textPayload)"
```

### Prisma Studio against production (read-only, careful)

Only do this if you need to inspect data for debugging. Never use Prisma Studio to modify production data.

```bash
# Start the Cloud SQL proxy first, then:
cd services/api
DATABASE_URL="postgresql://app_user:PROD_PASS@localhost:5432/ecommerce_builder" \
  npx prisma studio
```

## 7. Secret Rotation

All secrets live in Cloud Secret Manager. Secret Manager supports multiple versions — the app always reads `version=latest`.

### Rotate all secrets

```bash
cd infrastructure/scripts
./create-secrets.sh --project=YOUR_PROJECT_ID --rotate
```

This creates new secret versions for `db-password`, `jwt-secret`, and `encryption-key`. The script will print the new values — **save them before closing the terminal** and update `terraform.tfvars` and GitHub secrets accordingly.

After rotating:
1. Update `terraform.tfvars` with the new values (so Terraform doesn't revert them)
2. Update the corresponding GitHub secrets (`TF_DB_PASSWORD`, `TF_JWT_SECRET`, `TF_ENCRYPTION_KEY`)
3. Update the Cloud SQL user password to match the new `db-password`:
   ```bash
   gcloud sql users set-password app_user \
     --instance=ecommerce-builder-pg \
     --password=NEW_PASSWORD
   ```
4. Redeploy the API to pick up the new secret version:
   ```bash
   gcloud run services update ecommerce-api --region=us-central1
   ```
   This triggers a new revision that reads the latest secret versions.

### Rotate Gemini API key

```bash
echo -n "YOUR_NEW_KEY" | gcloud secrets versions add ecommerce-builder-gemini-api-key \
  --project=YOUR_PROJECT_ID --data-file=-
gcloud run services update ecommerce-api --region=us-central1
```

### Rotate the Terraform SA key (CI/CD)

```bash
# Create a new key
gcloud iam service-accounts keys create new-tf-sa-key.json \
  --iam-account=terraform-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com

# Update GCP_SA_KEY in GitHub repository secrets with the contents of new-tf-sa-key.json

# List old keys and delete the one you just replaced
gcloud iam service-accounts keys list \
  --iam-account=terraform-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com

gcloud iam service-accounts keys delete OLD_KEY_ID \
  --iam-account=terraform-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com
```

## 8. Scaling

### Cloud Run autoscaling

Cloud Run scales to zero by default. The Terraform config sets:
- `min_instance_count = var.min_instances` (default: 0)
- `max_instance_count = var.max_instances` (default: 10)

To prevent cold starts for the API, set `min_instances = 1` in `terraform.tfvars` and re-apply. This costs ~$15/month for a continuously running minimum instance.

```hcl
# terraform.tfvars
min_instances = 1   # warm instance always available
max_instances = 20  # increase if needed
```

### Cloud SQL scaling

The default tier is `db-f1-micro` (1 vCPU, 614 MB RAM) — suitable for development and low traffic. For production:

```hcl
# terraform.tfvars
db_tier = "db-n1-standard-2"   # 2 vCPU, 7.5 GB RAM
```

Disk autoresize is enabled — it will expand automatically. It does **not** shrink automatically.

**Scaling vertically requires a brief restart** (Cloud SQL will restart the instance). Plan for ~1-2 minutes of downtime. Schedule it during the maintenance window (Sundays 3 AM UTC).

### Read replicas

Not provisioned by default. To add one for analytics queries that don't hit the primary:
```bash
gcloud sql instances create ecommerce-builder-pg-read \
  --master-instance-name=ecommerce-builder-pg \
  --tier=db-n1-standard-1 \
  --region=us-central1
```

## 9. Runbooks

### RB-01: Cloud Run service is returning 5xx

1. Check Cloud Run logs:
   ```bash
   gcloud logging read 'resource.type="cloud_run_revision" \
     resource.labels.service_name="ecommerce-api" severity>=ERROR' \
     --limit=50 --format="table(timestamp,textPayload,jsonPayload)"
   ```

2. Check if the issue started with a recent deploy:
   ```bash
   gcloud run revisions list --service=ecommerce-api --region=us-central1
   ```

3. If a bad deploy, roll back to the previous revision:
   ```bash
   # Get the previous revision name from the list above
   gcloud run services update-traffic ecommerce-api \
     --to-revisions=PREVIOUS_REVISION=100 --region=us-central1
   ```

4. If not deploy-related, check if Cloud SQL is healthy:
   ```bash
   gcloud sql instances describe ecommerce-builder-pg --format="value(state)"
   # Should return RUNNABLE
   ```

5. Check Secret Manager access (secrets inaccessible = startup failure):
   ```bash
   gcloud secrets versions access latest --secret=ecommerce-builder-jwt-secret
   ```

---

### RB-02: Cloud SQL is slow or CPU is high

1. Check Query Insights in GCP Console (SQL → ecommerce-builder-pg → Query Insights)
2. Identify the top CPU-consuming queries
3. For N+1 queries or missing indexes, work with a developer — do not modify the schema directly
4. For a runaway query, kill it:
   ```bash
   # Connect via Cloud SQL proxy
   psql "host=localhost dbname=ecommerce_builder user=app_user password=PASS"

   -- Find long-running queries
   SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
   FROM pg_stat_activity
   WHERE (now() - pg_stat_activity.query_start) > interval '5 minutes';

   -- Kill it
   SELECT pg_terminate_backend(pid);
   ```
5. If CPU is consistently high, scale the Cloud SQL tier (see Section 8)

---

### RB-03: Disk space warning on Cloud SQL

Disk autoresize is enabled — it expands automatically when 90% full. No manual action needed unless autoresize is failing.

If autoresize has failed:
```bash
gcloud sql instances patch ecommerce-builder-pg --storage-size=50GB
```

To check current disk usage:
```bash
gcloud sql instances describe ecommerce-builder-pg \
  --format="value(settings.dataDiskSizeGb)"
```

---

### RB-04: CI/CD deploy is failing

1. Check the GitHub Actions run for the failing job — the error is almost always in the job logs
2. Common causes:
   - **Image push failed**: GCR permissions on `terraform-deployer` service account — verify `roles/storage.admin` is still assigned
   - **Migration job failed**: Check migration job logs (see Section 6). Usually a schema conflict or syntax error in a new migration
   - **Cloud Run update rejected**: Terraform may have changed resource configuration — try deploying manually first to isolate the issue
   - **Smoke test failed**: The API is returning non-200 on `/health`. Follow RB-01

3. If you need to deploy outside of CI:
   ```bash
   gcloud run services update ecommerce-api \
     --image=gcr.io/YOUR_PROJECT/ecommerce-api:COMMIT_SHA \
     --region=us-central1
   ```

---

### RB-05: Secret Manager access denied at runtime

Symptoms: API starts but returns 500 on all authenticated routes; logs show `Error: 7 PERMISSION_DENIED`.

1. Verify the service account still has `roles/secretmanager.secretAccessor`:
   ```bash
   gcloud projects get-iam-policy YOUR_PROJECT \
     --flatten="bindings[].members" \
     --filter="bindings.members:ecommerce-builder-app" \
     --format="table(bindings.role)"
   ```

2. If the role is missing (can happen if someone ran `terraform destroy` on just the IAM resources):
   ```bash
   gcloud projects add-iam-policy-binding YOUR_PROJECT \
     --member="serviceAccount:ecommerce-builder-app@YOUR_PROJECT.iam.gserviceaccount.com" \
     --role="roles/secretmanager.secretAccessor"
   ```

3. Restart the Cloud Run service to force a fresh startup:
   ```bash
   gcloud run services update ecommerce-api --region=us-central1
   ```

---

### RB-06: VPC Connector is over capacity

Symptoms: Cloud Run API is slow; logs show VPC connector throughput near maximum.

1. Check connector usage:
   ```bash
   gcloud compute networks vpc-access connectors describe ecommerce-connector \
     --region=us-central1
   ```

2. Scale up the connector:
   ```bash
   # Edit infrastructure/terraform/main.tf
   # Change max_instances from 3 to 10
   terraform apply
   ```

   Note: VPC connector scaling takes a few minutes and does not cause downtime.

## 10. Disaster Recovery

### RTO / RPO targets (defaults)

| Scenario | Recovery Time Objective | Recovery Point Objective |
|---|---|---|
| Cloud Run service failure | ~2 min (autoscale + health checks) | 0 (stateless) |
| Database corruption | ~30 min (restore + migrate) | Up to 1 day (daily backup) or seconds (PITR) |
| Accidental data deletion | ~15 min (PITR to pre-deletion point) | Seconds |
| Full region failure | Manual (no multi-region by default) | Daily backup |

### Full platform restore procedure

If the entire project is lost or corrupted:

1. **Re-bootstrap GCP:**
   ```bash
   ./infrastructure/scripts/setup-gcp.sh --project=NEW_PROJECT_ID
   ```

2. **Restore terraform.tfvars** from your backup (the secrets it contains are critical)

3. **Apply Terraform:**
   ```bash
   cd infrastructure/terraform && terraform apply
   ```

4. **Restore the database** from the last Cloud SQL backup or export:
   ```bash
   # If restoring from a Cloud SQL backup on the same instance:
   gcloud sql backups restore BACKUP_ID --restore-instance=ecommerce-builder-pg

   # If starting fresh from a SQL dump:
   gcloud sql import sql ecommerce-builder-pg gs://YOUR_BACKUP_BUCKET/dump.sql \
     --database=ecommerce_builder
   ```

5. **Push images and run migrations** (see Section 3d and 3e)

### Exporting a SQL dump (periodic, store offsite)

```bash
gcloud sql export sql ecommerce-builder-pg \
  gs://YOUR_PROJECT-tf-state/db-backups/$(date +%Y-%m-%d).sql \
  --database=ecommerce_builder \
  --offload
```

Run this as a scheduled Cloud Scheduler job for regular offsite backups beyond the 7-day automatic window.

## 11. Cost Management

Approximate monthly costs at low traffic:

| Service | Config | Est. Cost/mo |
|---|---|---|
| Cloud SQL | db-f1-micro, 20 GB | ~$10 |
| Cloud Run | 4 services, scale-to-zero | ~$5-20 |
| VPC Connector | 2 min instances | ~$15 |
| Cloud Storage | ~10 GB media | ~$0.20 |
| Secret Manager | 6 secrets | ~$0.30 |
| Total | | **~$30-50** |

**Biggest cost levers:**
1. **Cloud SQL tier** — `db-f1-micro` is cheap; `db-n1-standard-4` is ~$150/mo. Only upgrade when needed.
2. **VPC Connector** — `min_instances = 2` (the minimum) costs ~$15/mo regardless of traffic. Cannot be reduced further.
3. **Cloud Run min instances** — setting `min_instances = 0` (default) eliminates instance-hours cost at the expense of cold starts.

**To check current spend:**
```bash
gcloud billing budgets list --billing-account=YOUR_BILLING_ACCOUNT
```

Set a billing budget alert in GCP Console (Billing → Budgets & Alerts) to notify you if spend exceeds a threshold.

## 12. Security Posture

### What is in scope for you as SRE

| Item | Status |
|---|---|
| Cloud SQL has no public IP | Enforced by Terraform (`ipv4_enabled = false`) |
| Secrets stored in Secret Manager | All sensitive values, no plaintext in Cloud Run config |
| Service account follows least privilege | Defined in Terraform, review quarterly |
| GCS bucket media is public read, not write | `objectViewer` for `allUsers`, `objectAdmin` for SA only |
| TF SA key rotation | Rotate the `GCP_SA_KEY` GitHub secret every 90 days |
| Cloud Run services use HTTPS only | Enforced by GCP — HTTP is redirected automatically |
| Database encryption at rest | Enabled by default on Cloud SQL (Google-managed keys) |

### Audit log review

Enable data access audit logs for Cloud SQL and Secret Manager if you need compliance evidence:
```bash
gcloud projects get-iam-policy YOUR_PROJECT > policy.yaml
# Add auditConfigs for cloudsql.googleapis.com and secretmanager.googleapis.com
# Then: gcloud projects set-iam-policy YOUR_PROJECT policy.yaml
```

### Vulnerability scanning

Container images in GCR are automatically scanned by Container Analysis (Artifact Analysis) for known CVEs. View results:
```bash
gcloud artifacts docker images list gcr.io/YOUR_PROJECT \
  --show-occurrences --occurrence-filter="kind=VULNERABILITY"
```

The `test.yml` CI workflow also runs `npm audit --audit-level=high` for each package. Dependabot can be enabled on the GitHub repository to automate dependency updates.

### Terraform state security

The state file in `gs://{project}-tf-state/terraform/state` contains sensitive resource metadata. Ensure:
- The bucket has `uniform_bucket_level_access = true` (set by Terraform)
- Only the `terraform-deployer` service account has `objectAdmin` on it
- No one else has `roles/storage.objectViewer` on the bucket
